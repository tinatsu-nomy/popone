# ロードマップ

[English](ROADMAP.md)

このドキュメントは `popone` の計画中作業・将来改善・外部レビュー要望を追跡します。ここに記載された項目は **まだ実装されていません**。完了した項目は [CHANGELOG.jp.md](CHANGELOG.jp.md) に記録されます。

現在のターゲット: **v0.6.0**

> **注記:** v0.5.4 でスロット毎 UV 変形編集 UI を実装済み。v0.5.5 で頂点単位 UV エディタ **Phase 1 + Phase 2 + Phase 3 A-1 / A-3 / A-4 / A-5**（単一/複数頂点選択・ドラッグ、矩形選択（加算/除外モード含む）、ズーム/パン、回転/スケール、undo/redo、テクスチャ背景、独立 OS ウィンドウ化、UV0 / UV1 セット切替、2D ギズモハンドル）を実装済み（[CHANGELOG.jp.md](CHANGELOG.jp.md#v0552026-04-13) 参照）。残る Phase 3 の項目（モーフ UV）は v0.6.0 以降でのデータ対応拡大項目。

## v0.6.0 ターゲット

### UV エディタ Phase 3（対応データ拡大）

- **モーフ UV (A-2)** — PMX UV モーフに対して「ベース UV 編集 vs モーフ編集」モードスイッチ

### OBJ / STL インポートオプション UI の強化

インポートオプションダイアログは実装済みですが、座標系・単位プリセットを以下のように強化:

- ディレクトリごとに最後に使用したプリセットを記憶
- カスタムスケールフィールドを追加
- CLI の `--obj-unit` / `--stl-unit` フラグとして公開

### バックグラウンドロード内部整備

非同期ロードパイプラインの後続整備:

- **`CpuParseInput::ArchiveEntry` バリアント** — アーカイブ内ブラウズを BG 化する際に必要
- **`CpuParseInput::Reload` バリアント** — 現在個別処理されているリロードパスを統合

### ドラッグ＆ドロップによるスロット選択

材質編集ドロワー開時に画像ファイルを D&D した場合、スロット選択ダイアログを開く（現在の BaseColor 限定フローの代わりに）。これにより Emissive / Normal / Shade などにテクスチャを直接ドロップできる。

### 自動割当のスロットヒント

`auto_assign_textures()` は現在ファイル名と材質名のマッチのみ。ファイル名のスロット接尾辞ヒント（`*_normal*` → Normal、`*_emissive*` → Emissive 等）を認識するよう拡張し、PBR テクスチャセットの一括 D&D で正しいスロットに割り当てられるようにする。

### セクション折りたたみ状態の永続化

エディタドロワーのセクション折りたたみ状態は現在セッションごとにリセットされる。開いているセクション集合を `popone.toml` の `[material_editor] expanded_sections` に永続化する。

### ユーザーカスタムプリセット保存 / 読込

`MaterialPreset` は現在ビルトイン 3 種（MToon 1.0 / lilToon / PMX 互換）のみ。`CustomPreset { name: String, override_: MaterialParamOverride }` 型を定義して `popone.toml` に永続化し、エディタに「現在の値を保存...」ボタンを追加する。

### 材質編集ウインドウの複数同時表示

材質間のパラメータを並べて比較できるよう、エディタドロワーを同時に複数開けるようにする。

### 材質エディタの高度な MME 機能

- sdPBR `.fx` 生成
- MaskedMaterial（ray-mmd-MaskedMaterial）対応、マルチレイヤー合成
- MME `.fx` 読み込み（既存 PMX の `.fx` を解析してエディタにシード）

## 将来対応（ターゲット未設定）

### Unity `.anim` 残課題

`.anim` インポータは基本的なヒューマノイドアニメーションで動作しますが、以下の既知の不足があります:

1. **Muscle 角度範囲の精度** — 現在の `min_deg` / `max_deg` は推定値。Unity `HumanTrait.GetMuscleDefaultMin/Max()` が正確な Muscle 単位の範囲を提供（左右非対称なものもあり）
2. **末端ボーンの軸検証** — `muscle_vrm_axes()` 内の足首 / 手 / 指 Muscle の X/Y/Z 軸割り当てを Unity のリファレンスリグで厳密に検証していない
3. **Foot IK ゴール** — `LeftFootT/Q`・`RightFootT/Q` カーブを現在無視
4. **表情カーブ** — `.anim` の BlendShape カーブを未パース

### アーカイブ親ディレクトリ参照

一部のアーカイブは `../` の親ディレクトリパスでテクスチャを参照しています。現在の `sanitize_rel_path` は、ディスク読み込み時に `..` を除去します（テクスチャパストラバーサル対策のセキュリティ上の制約）。アーカイブ内での正当な `../` 参照をサポートするには:

- アーカイブルート境界内（安全）か、外に逃げる（危険）かの判定
- アーカイブコンテンツツリー内に限定して `..` を許可する別 sanitizer の追加

この対応が入るまで、該当アーカイブは警告を出してテクスチャ解決に失敗します。

### MTL サブディレクトリ解決

`.obj` がサブディレクトリの `.mtl` を参照している場合、`.mtl` から参照されるテクスチャは `.mtl` のディレクトリを基準に解決されるべきですが、現在は `.obj` のディレクトリを全パスに使用しています。

## コード品質

### GPU シェーダー自動テスト

WGSL シェーダーソースの自動検証が存在しません。推奨:

- **`naga` WGSL コンパイルテスト** — CI で全シェーダーの構文検証
- **Rust 側ライティング計算のユニットテスト** — `calc_lighting_mtoon_core` 等のヘルパーに、既知入力での決定的ユニットテストを追加

### 7z 展開 Phase 1 メモリピーク

7z 展開中、ライブラリは圧縮ソースと展開出力の両方を一時的に同時保持します。これは現在の 7z crate の構造的制約です。Phase 2（展開後）は圧縮バッファの即時解放で緩和済みですが、Phase 1（展開中）の解決には以下のいずれかが必要:

- mmap ベースの入力ソースで圧縮アーカイブ全体を RAM に読み込まないアプローチ
- 出力バッファリングを制限したストリーミング展開
